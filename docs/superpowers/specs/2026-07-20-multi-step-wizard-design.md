# Phase 9：多步骤向导设计

## 目标

在一次 InBridge 交互中，将复杂配置拆分为多个有明确语义的步骤。用户可以前进、返回和修改已填内容，只在最后一步统一确认并触发 ChatGPT 继续当前任务。

首个生产场景是将 `experiment_config` 拆分为“基础信息 → 训练配置 → 消融与确认”三步向导。

## 协议设计

保留现有扁平 `controls`，在交互配置顶层增加可选的 `steps`：

```json
{
  "controls": [
    { "id": "direction", "type": "select", "label": "研究方向" },
    { "id": "budget", "type": "range", "label": "训练预算" }
  ],
  "steps": [
    {
      "id": "basic",
      "title": "基础信息",
      "controlIds": ["direction"]
    },
    {
      "id": "training",
      "title": "训练配置",
      "controlIds": ["budget"]
    }
  ]
}
```

步骤结构：

```ts
interface InteractionStep {
  id: string;
  title: string;
  description?: string;
  controlIds: string[];
}
```

没有 `steps` 时，Widget 继续渲染现有单页表单。结果协议保持 `version: "1"`，现有 MCP 客户端和 ChatGPT 后续处理无需改变。

## 服务端约束

- `steps` 存在时必须包含 2–8 步。
- 步骤 `id` 必须唯一，每步至少包含一个控件。
- `controlIds` 只能引用当前配置中的真实控件。
- 每个控件必须恰好归入一个步骤；禁止遗漏和重复归属。
- 将各步 `controlIds` 依次展开后，必须与原始 `controls` 顺序完全一致。
- 条件控件继续只能引用自身之前的控件，因此不得依赖后续步骤。
- 步骤标题、说明和标识符执行长度与格式白名单校验。
- 不接受 HTML、JavaScript、CSS、表达式或外部 URL。

这些约束在 MCP 服务端完成。Widget 只消费已经验证的声明式配置。

## Widget 架构

向导状态包括：

```ts
interface WizardState {
  currentStepIndex: number;
  values: Record<string, unknown>;
}
```

控件值仍由现有 DOM 收集逻辑维护。步骤切换只改变哪些控件容器可见，不销毁控件节点，因此返回上一步时输入值自然保留。

向导相关的纯逻辑放入独立模块，负责：

- 判断配置是否启用向导；
- 获取当前步骤的控件集合；
- 计算步骤进度；
- 决定当前按钮是“下一步”还是最终提交；
- 限制上一步和下一步的边界。

`main.ts` 负责 DOM 渲染、当前步骤校验、按钮事件以及复用既有提交闭环。

## 交互行为

### 进度

向导顶部显示“步骤 N / M”以及全部步骤名称，并通过当前、已完成和待处理状态帮助用户定位。状态同时使用可访问属性表达，不只依赖颜色。

### 前进

“下一步”只校验当前步骤中可见且必填的控件。校验失败时停留在当前步骤，并将焦点移动到第一个错误字段。前进不调用模型，也不提交结果。

### 返回

第一步不显示“上一步”。其余步骤允许返回，所有输入和条件控件状态保留。返回不重复校验。

### 条件控件

`visibleWhen` 在所有步骤继续实时计算。隐藏字段不参与当前步骤校验，不进入摘要，也不进入最终结果；重新显示时恢复此前的本地值。

### 最终确认

最后一步展示完整的可见配置摘要，并将主按钮切换为模板的 `submitLabel`。点击后才调用 `updateModelContext` 与 `sendMessage`。防重复提交、重试和 JSON 复制降级路径全部沿用现有实现。

### 取消

配置提供 `cancelLabel` 时，每一步都允许取消。取消结果为 `status: "cancelled"` 且 `values: {}`，不基于未确认的默认值继续任务。

## 实验配置模板

`experiment_config` 使用以下三步：

1. 基础信息
   - `research_direction`
   - `environments`
   - `information_density`
2. 训练配置
   - `training_budget`
   - `seed_count`
   - `primary_color`
   - `note`
3. 消融与确认
   - `ablation`
   - `ablation_variables`

`ablation_variables` 仅在 `ablation` 开启时显示。完整摘要位于最后一步，但只包含当前可见字段。

## 错误处理

- 非法步骤配置由服务端拒绝，并返回可定位到字段的校验信息。
- UI 收到缺失或无效的步骤数据时显示现有初始化失败提示，不尝试猜测分页。
- 当前步骤必填项错误使用就地提示，不改变已有输入。
- 模型上下文或 follow-up message 失败时，继续使用现有重试与 JSON 回退流程。

## 测试策略

采用测试驱动开发，先观察每项新行为因功能缺失而失败，再编写最小实现。

服务端测试覆盖：

- 合法多步骤配置；
- 无 `steps` 的单页向后兼容；
- 重复步骤标识；
- 未知、重复或遗漏的控件引用；
- 控件乱序；
- 步骤数量和文本边界；
- `experiment_config` 的三步输出。

UI 纯逻辑与浏览器测试覆盖：

- 初始步骤与进度；
- 当前步骤可见性；
- 当前步骤必填校验；
- 前后切换与值保留；
- 条件控件显隐；
- 最后一步统一提交；
- 取消、防重复提交和失败降级不回归。

发布前运行类型检查、全部单元测试、构建、本地浏览器操作和生产环境冒烟测试。部署后由用户在 ChatGPT 中完成真实三步实验配置并确认模型准确读取最终结构化结果。

## 验收标准

1. 旧的一页式交互保持原有行为。
2. 合法配置可以声明 2–8 个步骤，非法引用在服务端被拒绝。
3. 用户只能在当前步骤必填项通过后前进。
4. 返回上一步不会丢失输入。
5. 条件隐藏字段不阻塞前进，也不进入摘要或最终结果。
6. 只有最后确认才向 ChatGPT 同步一次结果并触发下一轮。
7. `experiment_config` 在 ChatGPT 中按三个步骤正常完成。
8. ChatGPT 能读取全部最终选择并继续当前任务。

## 非目标

- 不实现模型调用驱动的逐步问答。
- 不实现步骤分支、跳步或任意流程图。
- 不持久化未提交的向导草稿。
- 不改变交互结果协议版本。
- 不为普通单页模板强制启用分页。
